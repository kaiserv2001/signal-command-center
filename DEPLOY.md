# SIGNAL — $0 Deployment Guide

SIGNAL ships to the public internet for **zero dollars**. Every layer below has a
genuine perpetual free tier (not a trial) and needs no credit card.

| Layer | Host | Free tier |
|-------|------|-----------|
| Database | MongoDB Atlas | M0 cluster — 512 MB, shared, free forever |
| Backend (Express + Socket.io) | Render | Free web service — supports WebSockets |
| Frontend (Vite static build) | Vercel | Free static hosting + global CDN |

The repo already contains everything the hosts need:
- `render.yaml` — Render Blueprint for the backend
- `client/vercel.json` — Vercel build + SPA rewrite for the frontend
- `server/.env.example`, `client/.env.example` — the env vars to set in each dashboard

---

## 0. Prerequisites
- A GitHub repo with this code pushed (Render + Vercel deploy from GitHub).
- Free accounts: [MongoDB Atlas](https://www.mongodb.com/atlas), [Render](https://render.com), [Vercel](https://vercel.com).

---

## 1. Database — MongoDB Atlas M0
1. Create a free **M0** cluster.
2. **Database Access** → add a user (username + password). Save the password.
3. **Network Access** → add IP `0.0.0.0/0` (allow from anywhere). Render's free tier
   has no static outbound IP, so this is required for the API to reach Atlas.
4. **Connect → Drivers** → copy the connection string. It looks like:
   `mongodb+srv://USER:PASSWORD@cluster0.xxxx.mongodb.net/signal?retryWrites=true&w=majority`
   Make sure the database name `signal` is in the path (before the `?`).

Keep this string — it's the `MONGODB_URI` for Render.

---

## 2. Backend — Render (Blueprint)
1. **New → Blueprint**, point it at this GitHub repo. Render reads `render.yaml`
   and provisions the `signal-api` web service.
2. Set the env vars Render marks as required (these are `sync: false` in the blueprint):
   | Key | Value |
   |-----|-------|
   | `MONGODB_URI` | the Atlas string from step 1 |
   | `CLIENT_ORIGIN` | your Vercel URL (fill in after step 3, then redeploy) |
   - `JWT_SECRET` is auto-generated by Render (`generateValue: true`).
   - `PORT` and `NODE_VERSION` are already set in `render.yaml`.
3. Deploy. When it's live, hit `https://<your-api>.onrender.com/api/health` →
   expect `{"ok":true,"service":"signal-api",...}`.

Note the API URL (e.g. `https://signal-api.onrender.com`) — it's the `VITE_API_URL` for Vercel.

---

## 3. Frontend — Vercel
1. **Add New → Project**, import the same repo.
2. Set **Root Directory** to `client/`. Vercel picks up `client/vercel.json`
   (framework `vite`, build `npm run build`, output `dist`, SPA rewrite).
3. Add an env var:
   | Key | Value |
   |-----|-------|
   | `VITE_API_URL` | the Render API URL from step 2 (no trailing slash) |
4. Deploy. Copy the resulting URL (e.g. `https://signal.vercel.app`).

---

## 4. Close the CORS loop
1. Back in Render, set `CLIENT_ORIGIN` to your Vercel URL and redeploy the API.
   - `CLIENT_ORIGIN` accepts a **comma-separated list** if you also want Vercel
     preview deployments to work, e.g.
     `https://signal.vercel.app,https://signal-git-main-you.vercel.app`.
2. Open the Vercel URL → log in (or **▶ ENTER AS DEMO OPERATOR**) → you should see
   `● LIVE`, live markers on the map, and the incident ticker streaming.

## Honest free-tier heads-up (this is a feature, not an apology)
- **Render cold starts.** The free web service sleeps after ~15 min idle; the first
  request then takes **30–60 s** to wake. For a portfolio demo this is fine — just
  know the first load after a quiet spell is slow, it isn't broken.
- **OpenSky rate limits** anonymous aircraft calls. SIGNAL caches each source and,
  if throttled, keeps serving the last-known data so the map never blanks. Cold
  starts briefly show fixture aircraft until the first live pull lands.
- **Atlas M0** is shared and capped (512 MB, limited connections) — great for a demo,
  not for production scale.
- **No separate background worker.** Render free has no always-on cron, so the feed
  refresh intervals run inside the web process — they tick while the service is awake
  (i.e. whenever someone's using it), which is exactly what a demo needs.
- **WebSockets** work on Render's free tier, so the live `feed:update` / `alert:hit`
  socket stream runs in production, not just locally.

## Local dev (one command)
```bash
npm run install:all      # root + server + client deps
cp server/.env.example server/.env   # fill in MONGODB_URI (or leave blank to run DB-less)
npm run dev              # server :4000 + client :5173 (Vite proxies /api + /socket.io)
```
Locally `VITE_API_URL` stays blank, so the browser talks to one origin through the
Vite proxy and there's no CORS to configure.
